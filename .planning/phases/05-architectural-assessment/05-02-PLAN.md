<action>
Create stability report evaluating the following areas. Use severity ratings: Critical/High/Medium/Low.

**1. Error Handling Completeness**

- Review all `#[tauri::command]` functions in lib.rs
- Check for `unwrap()` or `expect()` that could panic
- Verify `Result<T, String>` pattern is consistent
- Check frontend error handling (try/catch in invoke calls)

**Assessment areas:**

- Database connection failures
- File I/O errors (import/export)
- FTS rebuild failures
- Concurrent access issues

**2. Edge Case Coverage**

- Empty data: What happens when database is empty?
- Large datasets: Performance with 10k+ words? 100k+ definitions?
- Malformed input: What if LOD file is corrupted?
- Unicode handling: Non-ASCII characters in definitions?

**Check:**

- `get_words` with empty database
- `rebuild_fts` with large definition count
- `import_lod_contents` with invalid format
- Search with special characters

**3. Cross-Platform Consistency**

- Windows: Path handling, file dialogs
- Android: Content URIs, file system access
- Identify platform-specific code paths

**4. Database Integrity**

- Migration safety (from CONCERNS.md: no rollback)
- Foreign key constraints (enabled via PRAGMA)
- Concurrent access (Mutex usage)
- WAL mode usage for FTS rebuild

<action>
Create maintainability scorecard evaluating the following areas.

**1. Code Organization**

- Backend modules: `lib.rs`, `db.rs`, `models.rs`, `import.rs`, `export.rs`
- Are responsibilities clear?
- Is there a god object? (Check if lib.rs has too many responsibilities)
- Frontend components: Are they well-organized?

**Assessment:**

- Module boundaries: Clear or fuzzy?
- File sizes: Any files > 1000 lines? (lib.rs is ~903 lines)
- Cohesion: Do modules have single responsibility?

**2. Coupling Analysis**

- Frontend-backend coupling: How tight?
- Database coupling: Is db.rs called directly or through abstractions?
- State coupling: Is AppState accessed directly or through helpers?

**Check:**

- How many places call `state.db.lock()`?
- How many components import directly from types.ts?
- Is there shared mutable state beyond AppState?

**3. Documentation Coverage**

- Inline comments: Are complex sections explained?
- Module-level docs: Do files have `//!` or `///` comments?
- README: Is there build/run documentation?
- AGENTS.md: Is developer guide complete?

**Assessment:**

- Rate: Good/Adequate/Poor
- Identify undocumented complex areas

**4. Extensibility**

- Adding new word fields: How hard?
- Adding new entity type (e.g., "examples"): What files change?
- Adding new import format: How much code?
