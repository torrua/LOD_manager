- Error handling: Is `?` operator used? Is `map_err` consistent?

- Indexing: Are indexes present on query columns? (Check `idx_*` in schema)
- FTS5: Is FTS properly configured with content table linking?
- Foreign keys: Are constraints enabled? (`PRAGMA foreign_keys=ON`)
- Migrations: Are they safe and idempotent?

<action>
Document design patterns in use and evaluate appropriateness.

**1. Identify Patterns in Use**

**Command Pattern** (Tauri invoke):

- Location: `src-tauri/src/lib.rs` - all `#[tauri::command]` functions
- Assessment: Is this appropriate for a desktop app? Does it separate concerns well?
- Example: `open_database`, `get_words`, `save_word`, etc.

**Repository Pattern** (Database operations):

- Location: `src-tauri/src/db.rs` - functions like `list_words`, `get_word`, `save_word`
- Assessment: Is there separation between data access and business logic?
- Example: `db::list_words()` wraps SQL queries

**State Pattern** (AppState):

- Location: `src-tauri/src/lib.rs` - `AppState` struct with `Mutex<Option<Connection>>`
- Assessment: Is global state appropriate? Are there alternatives?

**Observer Pattern** (Svelte reactivity):

- Location: Frontend stores in `src/lib/store.svelte.ts`
- Assessment: Are runes used correctly? Is state management clear?

**2. Identify Anti-Patterns**
Check for:

- God objects (single file with too many responsibilities)
- Tight coupling (imports that shouldn't exist)
- Primitive obsession (using strings where types would be better)
- Magic numbers/strings without constants

**3. Pattern Appropriateness for Domain**

- This is a dictionary manager (~28 commands), not enterprise software
- Are patterns over-engineered or appropriate?
- What patterns are missing that would help?
